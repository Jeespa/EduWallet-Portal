# EduWallet Portal Backend

The EduWallet Portal Backend is the server-side API used by the EduWallet Portal. It is separate from the student-facing gateway.

The gateway is used by student clients such as the mobile app and browser extension. The portal backend is used by organizations such as universities, employers, and external verifiers. It handles portal users, organization roles, request history, verification history, issuance drafts, and portal-side persistence.

In the local demo setup, the backend uses generated demo student metadata to find student IDs and smart account addresses. Academic records and permission-related operations are then handled through the EduWallet SDK and smart contracts. PostgreSQL stores portal-side data such as users, organizations, request logs, verification logs, and issuance drafts.

## Current scope

The backend currently supports:

- Portal user authentication with JWT.
- Organizations, memberships, and portal roles.
- Student lookup through the generated demo student source.
- Permission request creation, listing, and status tracking.
- Verification creation and listing.
- Issuance draft creation, listing, and submission.
- PostgreSQL persistence for portal-side data.
- Local demo integration with the EduWallet contracts and SDK.

The backend does not implement public organization self-registration. In the thesis prototype, organizations and users are seeded for the demo. Real registration would require external identity verification before administrator accounts are issued.

## Project structure

```text
portal-backend/
  prisma/
    schema.prisma              # Prisma schema and migrations
  src/
    auth/                      # Authentication helpers and middleware
    generated/                 # Generated Prisma client output
    lib/                       # Shared backend utilities
    routes/                    # Express route modules
    scripts/                   # Seed script
    services/                  # Portal business logic
    students/                  # Demo student source abstraction
    types/                     # Backend type definitions
    index.ts                   # Express app entry point
```

`src/generated/prisma/` is generated by Prisma and should not be committed. Regenerate it with `npm run prisma:generate`.

## Prerequisites

- Node.js and npm.
- PostgreSQL.
- A local EduWallet demo chain if you want to run the full portal demo.
- The SDK built from the repository root.

For the full demo flow, start from the root README first. The root setup runs the local Hardhat chain and generates the demo blockchain files used by this backend.

## Environment configuration

Create a `.env` file in `portal-backend/`.

A minimal local development setup looks like this:

```env
PORT=4000
DATABASE_URL="postgresql://postgres:YOUR_PASSWORD@localhost:5432/eduwallet_portal"
JWT_SECRET="change-this-in-real-use"
STUDENT_SOURCE=demo
```

For the full local demo, `scripts/bootstrapPortalDemo.ts` also generates demo-chain environment files. Use the generated `portal-backend/.env.demo-chain` values together with the database and JWT settings above.

The expected student source for the cleaned demo setup is:

```env
STUDENT_SOURCE=demo
```

Other student source values are not supported by the current portal backend.

## Installation

From `portal-backend/`:

```bash
npm install
```

Generate the Prisma client:

```bash
npm run prisma:generate
```

Run migrations:

```bash
npm run prisma:migrate
```

Seed the database:

```bash
npm run seed
```

You can also run the full database setup with:

```bash
npm run db:setup
```

This runs Prisma generation, migrations, and the seed script.

## Running the backend

Start the development server:

```bash
npm run dev
```

The backend listens on:

```text
http://localhost:4000
```

Check that it is running:

```text
GET http://localhost:4000/health
```

Expected response:

```json
{
  "ok": true,
  "service": "portal-backend"
}
```

Build the backend:

```bash
npm run build
```

Run the compiled backend:

```bash
npm start
```

## Database commands

Generate Prisma client:

```bash
npm run prisma:generate
```

Create or update local development migrations:

```bash
npm run prisma:migrate
```

Apply existing migrations without creating new ones:

```bash
npm run prisma:deploy
```

Open Prisma Studio:

```bash
npm run prisma:studio
```

Reset the database and seed it again:

```bash
npm run db:reset:seed
```

Use the reset command with care. It deletes local portal backend data.

## Example PowerShell test flow

### Login

```powershell
$login = Invoke-RestMethod `
  -Method POST `
  -Uri "http://localhost:4000/auth/login" `
  -ContentType "application/json" `
  -Body '{"email":"ingrid@ntnu.no","password":"password123"}'

$token = $login.token
$login | ConvertTo-Json -Depth 10
```

### Current session

```powershell
Invoke-RestMethod `
  -Method GET `
  -Uri "http://localhost:4000/auth/me" `
  -Headers @{ Authorization = "Bearer $token" } |
  ConvertTo-Json -Depth 10
```

### Search students

```powershell
Invoke-RestMethod `
```

```powershell

  -Uri "http://localhost:4000/requests" `
  -Headers @{ Authorization = "Bearer $token" } |
  ConvertTo-Json -Depth 10
```

## Local demo notes

The portal backend depends on both PostgreSQL and the local EduWallet blockchain demo.

A typical full demo order is:

```text
1. Start the local Hardhat node from the repository root.
2. Run the demo bootstrap script from the repository root.
3. Configure portal-backend/.env using the generated demo-chain values.
4. Run PostgreSQL.
5. Run portal backend migrations and seed.
6. Start the portal backend.
7. Start the EduWallet Portal frontend.
```

If the Hardhat node is stopped, the local blockchain state is lost. Run the bootstrap script again and refresh the generated `.env.demo-chain` files.

## Prototype limitations

This backend is a thesis prototype and local demo backend. Important limitations:

- Authentication is suitable for local demo use only.
- Demo users and organizations are seeded, not registered through a production onboarding process.
- The student source is generated from the local demo setup.
- The backend is not production-hardened.
- The local Hardhat blockchain is temporary and must be recreated when reset.
- PostgreSQL stores portal-side state only. Academic records and permissions are represented in the EduWallet contracts.
